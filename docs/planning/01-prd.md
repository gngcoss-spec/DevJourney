# PRD - DevJourney

## MVP Capsule

1. **목표:** 혼자 8개 서비스를 동시 개발하면서, 서비스 전환 시 맥락 손실 없이 개발 여정을 관리하는 웹 도구
2. **페르소나:** AI를 활용해 다수의 서비스를 동시 개발하는 1인 개발자 (PM/PO 경험 보유)
3. **핵심 기능:** FEAT-1 Dashboard (전체 현황), FEAT-2 Service + Work Item (작업/의사결정 통합), FEAT-3 Dev Log (일일 개발 로그)
4. **성공 지표 (노스스타):** 서비스 전환 후 작업 재개까지 걸리는 시간 (목표: 5분 이내)
5. **입력 지표:** 주간 Dev Log 작성 수, Work Item 의사결정 필드 완성률
6. **비기능 요구:** 페이지 로드 2초 이내, 모바일 반응형 지원
7. **Out-of-scope:** 문서 버전 관리, 팀 권한 세분화, CI/CD 연동, Roadmap, Sprint
8. **Top 리스크:** 기록 부재로 인한 맥락 손실 반복
9. **완화/실험:** "레슨런" 관점으로 기록 - 다음에 꼭 필요한 정보(서버 설정법, 도구 설치법 등)를 기록하는 실용적 동기 활용 + 템플릿 자동 채우기
10. **다음 단계:** DB 스키마 설계 후 Supabase 프로젝트 생성

## 1. 제품 개요

### 1.1 제품명
**DevJourney** (가칭)

### 1.2 비전
"혼자 쓰기 시작하지만, 팀이 붙어도 그대로 개발 조직의 공식 기록이 되는 개발관리 웹"

### 1.3 제품 목적
- 여러 서비스 동시 개발 관리
- 개발 여정(의사결정 + 작업 + 변경 이력) 기록
- 서비스 전환 시 맥락 손실 방지
- 향후 팀원 온보딩 시 개발 히스토리 제공
- AI 기반 개발 기록(ChatGPT/Gemini 대화 세션) 포함

### 1.4 핵심 설계 원칙
1. **서비스 중심 구조** - 모든 정보는 서비스(프로젝트) 아래 조직됨
2. **의사결정 기록 우선** - 작업보다 "왜 그렇게 결정했는지"가 중요
3. **타임라인 항상 존재** - 개발 여정이 시간 순으로 추적 가능
4. **최소 입력 기록** - 부담 없이 적을 수 있어야 함
5. **Jira처럼 보이되, 과도하게 복잡하지 않음**

## 2. 사용자 문제 정의

### 2.1 핵심 고통
8개 서비스를 동시 개발하면서, 서비스 간 전환 시 발생하는 3가지 문제:

| 문제 | 설명 | 심각도 |
|------|------|--------|
| **진행 상황 파악 불가** | "마지막에 어디까지 했는지 모르겠다" | 높음 |
| **의사결정 맥락 손실** | "왜 그렇게 결정했는지 기억이 안 난다" | 높음 |
| **다음 액션 불명확** | "다음에 뭘 해야 하는지 정리가 안 된다" | 높음 |

### 2.2 추가 고통
- AI(ChatGPT/Gemini)와 나눈 대화에서 중요한 문제 해결 과정을 다시 찾기 어려움
- 개발 문서에 정의되지 않은 내용이 AI 대화로 해결되지만, 이 기록이 흩어져 있음

### 2.3 현재 해결 방식
- 없음 (기억에 의존)
- 결과: 서비스 전환 시 이전 단계를 이해하는 데 많은 시간 소요

## 3. 핵심 기능 (MVP)

### FEAT-1: Dashboard
**목적:** 8개 서비스의 전체 현황을 한눈에 파악

**구성:**
- 요약 카드 (전체 서비스 수, 진행중, 정체, 중단 위험)
- 마일스톤 그래프 + 진행률
- 서비스 카드 목록 (이름, 상태색상, 진행률, 단계, 마지막활동일, 다음액션)
- 통합 Dev Log 탭

**상태 색상:**
- 정상 진행 (녹색)
- 정체 - 7일 이상 활동 없음 (노란색)
- 중단 위험 (빨간색)

### FEAT-2: Service + Work Item (작업/의사결정 통합 관리)
**목적:** 서비스 등록 + 작업 관리 + 의사결정 기록을 통합

**Service CRUD:**
- 서비스명, 설명, 목표, 주요 사용자, 현재 단계, 현재 서버, 기술 스택, ChatGPT 역할 정의
- 8개 필드 전체 입력

**Work Item:**
- Kanban 보드 (5컨럼: Backlog / Ready / In Progress / Review / Done)
- 드래그앤드롭 상태 변경
- 빠른 생성 (컨럼 상단 + 버튼)
- 상세 모달 (4탭):
  - 기본 정보: 제목, 설명, 유형(Feature/Bug/Refactor/Infra/AI Prompt), 우선순위, 담당자, 상태
  - 의사결정: Problem(왜 필요한가), 고려한 선택지, 최종 결정 이유, 결과
  - AI 세션: AI 대화 링크 + 요약 목록
  - 활동 로그: 상태 변경 이력, 커멘트

### FEAT-3: Dev Log (일일 개발 로그)
**목적:** 일일 개발 기록을 최소 입력으로 작성

**작성 방식:**
- 템플릿 자동 채우기 + 자유 입력
- 4개 필드: 오늘 한 것 / 확정한 것 / 보류한 것 / 다음에 할 것

**표시 방식:**
- 서비스별: Service Detail의 Dev Logs 탭 (타임라인 형태)
- 통합: Dashboard의 통합 로그 탭

## 4. 정보 구조 (IA)

```
Dashboard (통합 현황 + 통합 로그 탭)
├─ Services
│    └─ Service Detail
│          ├─ Overview (상세 그래프 + 목적 + 단계 + 다음 액션)
│          ├─ Board (5컨럼 Kanban)
│          ├─ Work Items (목록 + 필터)
│          └─ Dev Logs (타임라인 + 템플릿 입력)
└─ (2차) Servers / Team / Activity Timeline / Documents
```

## 5. 벤치마킹

### 유사 서비스 분석

| 서비스 | 강점 | 약점 | DevJourney 차별점 |
|--------|------|------|-------------------|
| **Jira** | 강력한 이슈 트래킹, 스프린트 | 과도하게 복잡, 1인 개발자에겐 부담 | 복잡도 90% 감소, 의사결정 기록 내장 |
| **Linear** | 깔끔한 UI, 빠른 속도 | 의사결정 기록 없음, 다중 서비스 관리 약함 | AI 세션 연결, 서비스 중심 구조 |
| **Notion** | 유연한 구조, 문서 관리 | 프로젝트 관리 기능 약함, 칸반 제한적 | 전용 Kanban + 의사결정 + Dev Log 통합 |

### DevJourney가 "낫다"고 할 수 있는 이유
1. **AI 대화 세션 연결**: 다른 도구에는 없는 기능. AI와 문제 해결한 여정 추적
2. **의사결정 기록 내장**: Work Item 안에 "왜 이렇게 결정했는지"가 구조적으로 포함
3. **서비스 전환 최적화**: 다음 액션이 Dashboard와 Overview에서 즉시 확인 가능

## 6. 가설 vs 사실

| 구분 | 내용 | 검증 방법 |
|------|------|----------|
| **사실** | 8개 서비스 동시 개발 중 | 현재 상황 |
| **사실** | 서비스 전환 시 맥락 손실 발생 | 직접 경험 |
| **사실** | AI 대화로 개발 문제 해결 중 | 직접 경험 |
| **가설** | Dev Log 작성이 습관화될 것 | MVP 사용 후 2주간 작성률 측정 |
| **가설** | 의사결정 필드가 실제로 채워질 것 | MVP 사용 후 완성률 측정 |
| **가설** | 서비스 전환 시간이 5분 이내로 줄어들 것 | 전환 전/후 시간 비교 |

## 7. 타협 기록

| 타협 항목 | 포기한 것 | 유지한 것 | 이유 |
|----------|----------|----------|------|
| MVP 범위 | Sprint, Roadmap, Server, Timeline | Dashboard, Service, Work Item, Dev Log | 핵심 고통 해결에 집중 |
| Decision 관리 | 별도 Decision 엔티티 | Work Item 내 의사결정 필드 | MVP 복잡도 감소, 충분한 기록 가능 |
| 인증 | 소셜 로그인, Magic Link | 이메일 + 비밀번호 | 최소 구현 |
| 팀 기능 | 역할/권한 관리 | 단일 사용자 | "혼자 시작" 원칙 |

## 8. 실험 계획

### MVP 실험
- **기간:** 2주 개발 + 2주 사용
- **성공 기준:**
  - 주 5일 이상 Dev Log 작성
  - Work Item 의사결정 필드 70% 이상 완성
  - 서비스 전환 후 5분 이내 작업 재개

### 실패 시나리오
- **시나리오 1:** Dev Log를 안 씀 → 자동 리마인더 추가, 더 간단한 형식 도입
- **시나리오 2:** 의사결정 필드를 안 채움 → 필수/선택 구분 재검토, AI 자동 요약 도입
- **시나리오 3:** Dashboard가 쓸모없음 → Overview에서 직접 작업, Dashboard 제거

## 9. MVP 범위 vs 2차 개발

### MVP (Phase 1)
- 로그인/회원가입
- Dashboard (요약 + 그래프 + 서비스 목록 + 통합 로그)
- Service CRUD (8개 필드)
- Work Item (Kanban + 의사결정 + AI 세션)
- Dev Log (서비스별 + 통합 뷰)

### 2차 개발 (Phase 2)
- Sprint 계획 (전체 + 서비스별)
- Decision 별도 관리
- Server 관리
- Activity Timeline
- 문서 관리
- 팀 관리 (Owner/Contributor/Viewer)
- Roadmap (단계 기반 타임라인)
